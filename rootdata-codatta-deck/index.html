01.2 · 三大价值
价值问题缺陷基础方案
Three Pillars of Value

RootData 数据资产的
三个独特价值点

第一

Web3 场景明确

用户查询不止看简介——是在判断项目是否真实、什么阶段、谁投资、Token 与链上地址是否可信、有无审计或安全风险。

第二

结构化基础扎实

不是普通文章,而是可拆解为一条条 Claim:某项目某时间完成某轮融资 / Token 合约地址 / 与某生态合作 / 发生过某安全事件。

第三

AI Agent 数据底座

未来从"人查网页"变为"Agent 自动查询、比较、引用"。Agent 需要的不是页面文本,是带来源、可信度、验证状态的数据上下文。

因此,RootData 的长期价值不只是"Web3 项目资料库", 而是有机会升级为 Web3 项目可信事实层 —— 服务人类用户、机构客户、开发者和 AI Agent。

第二部分PART 02
02.1 · 问题 1–3
价值问题缺陷基础方案
5 Structural Problems · 1 / 2

五个绕不开的结构性问题

问题 01

信息来源分散,
真实性难判断

官网、白皮书、Twitter、Mirror、Discord、GitHub、链上交易、机构披露、审计报告、媒体报道——口径不一、过期信息仍被引用、链上链下需要交叉验证。

问题 02

数据更新和
维护成本高

Web3 项目变化快,完全依赖中心化编辑团队会面临覆盖范围、更新速度、审核成本和争议处理的压力。需要从"内部维护为主"升级为"内部把关 + 外部贡献 + 机制化验证"。

问题 03

贡献者身份和
贡献价值难沉淀

普通表单提交,贡献者没有长期身份,贡献也无法积累为信誉或资产。一次有效贡献应被记录:谁提交 / 内容是什么 / evidence / 是否通过验证 / 是否进入 lineage。

问题 1 — 3 / 5PART 02
02.2 · 问题 4–5
价值问题缺陷基础方案
5 Structural Problems · 2 / 2

缺乏结构化追溯与
Agent 可信上下文

问题 04 · LINEAGE

数据来源、版本和争议
缺少结构化追溯

项目资料持续变化,不适合简单覆盖。更合理的方式:

新提交进入候选版本 → 通过验证后成为公开版本 → 历史版本保留 → 冲突 claim 标记争议而非删除 → 重要信息保留 evidence → accepted claim 生成 lineage。

这样 RootData 不只展示"当前结果",而是能展示"这个结果如何形成"。

问题 05 · AGENT CONTEXT

AI / Agent 消费
需要可信上下文

普通 API 能告诉 Agent 一个字段值,但不能告诉 Agent 这个值为什么可信。

Agent 需要的不只是 project_name / funding_amount / token_address / investors,而是同时知道:

source · evidence_type · verification_status · trust_level · version · lineage_status · updated_at · dispute_status

从"字段返回"升级为"带可信上下文的数据返回"。

Step 1 — 3

STEP 01

Claim 化

项目融资 / Token 地址 / 团队成员 / 合作关系 / 审计 / 安全事件,统一建模为 Claim:
主体 + 类型 + 内容 + evidence + contributor + timestamp + status + version

STEP 02

贡献结构化

项目方 / 社区 / 研究员 / 编辑团队均可提交。Phase 1 不强制全部上链、不强制 DID。先跑通"用户提交→候选→验证→公开"业务闭环。

STEP 03

验证机制化

每条 claim 进入 Verification 流程:AI gate · 必填 / 格式 / 重复 / 风险检查 · 合约地址 / 交易哈希可查 · 少量人工复核。结果 = Accepted / Rejected / Needs Review。

RootData 编辑团队仍是最终把关方,但"审核通过"从一次后台修改, 变成一条可追溯的 verification 记录。
