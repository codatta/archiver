# AI Team Workflow 协议 v3

**这是主要的功能开发流程。** 用于所有非琐碎的新功能开发。

---

## 触发机制（Lead智能判断）

**用户描述任务后，Lead自动分析复杂度，决定走哪个流程。**

### Lead的复杂度评估清单

用户提出任务后，Lead分析以下5个维度：

1. **是否涉及UI设计？**（新页面、新组件、布局调整、原型需求）
2. **是否需要产品澄清？**（需求模糊、多种实现方式、边界不清、根因不明的bug）
3. **是否涉及多个文件？**（>3个文件修改）
4. **是否有重大影响？**（新功能、架构调整、核心逻辑重构、影响多模块的bug）
5. **是否可能有冲突？**（与现有功能可能重叠）

### 判断规则

| 符合项数 | 复杂度 | Lead行为 |
|---------|-------|---------|
| 0-1项 | 简单 | 直接实现（普通流程） |
| 2-3项 | 中等 | **询问用户**走哪个流程 |
| 4-5项 | 复杂 | **自动启动**AI Team Workflow |

### 四个流程级别（Lead自动判断）

#### Level 1: 直接实现（3-5分钟，0 agents）

**适用：** 纯文案/样式/配置，<10行改动，单文件

**流程：**
```
Lead直接修改代码 → git commit → git push → 创建PR
```

**跳过：** 全部agent

**Token消耗：** ~0.5k（Lead判断 + 代码修改 + PR创建）

---

#### Level 2: 简化流程（10-12分钟，3 agents简化版）

**适用：** 单文件小功能，10-50行，有参考实现，不需要UI原型

**流程：**
```
PM简化 → Dev → QA简化 → PR
```

**PM简化版：**
- 不做全repo探索（只看相关文件）
- 参考历史PRD模板
- 输出1-2页PRD

**QA简化版：**
- 只检查改动文件
- 简化回归评估（只列直接依赖）
- 验收清单只含主流程

**跳过：** UI原型、Pre-QA

**Token消耗：** ~4k（PM 1k + Dev 2k + QA 1k）

---

#### Level 3: 标准流程（18-20分钟，4 agents）

**适用：** 中等功能，50-200行，需要UI设计，有历史参考

**流程：**
```
PM → UI → Dev → QA → PR
```

**跳过：** Pre-QA（因为有历史参考，风险可控）

**Token消耗：** ~8k（PM 2k + UI 2k + Dev 2k + QA 2k）

---

#### Level 4: 完整流程（25-30分钟，5 agents）

**适用：** 复杂新功能，>200行，无历史参考，高风险

**流程：**
```
PM → UI → Pre-QA → Dev → QA → PR
```

**全部节点：** 无跳过

**Token消耗：** ~13k（PM 3k + UI 3k + Pre-QA 1k + Dev 4k + QA 2k）

### Lead判断逻辑

**判断维度（6个）：**

| 维度 | Level 1 | Level 2 | Level 3 | Level 4 |
|------|---------|---------|---------|---------|
| 改动规模 | <10行 | 10-50行 | 50-200行 | >200行 |
| 文件数 | 1个 | 1-2个 | 2-4个 | >4个 |
| UI设计 | 否 | 否 | 是 | 是 |
| 产品澄清 | 否 | 否 | 可能 | 是 |
| 历史参考 | - | 是 | 是 | 否 |
| 复杂度 | 低 | 低 | 中 | 高 |

**判断流程：**

```python
def judge_workflow_level(task):
    # 1. 快速排除Level 1
    if is_pure_text_or_style(task) and estimate_lines < 10:
        return "Level 1"
    
    # 2. 检查是否有历史参考
    has_reference = search_similar_features(task)
    
    # 3. 预估改动规模
    estimated_lines = estimate_code_changes(task)
    estimated_files = estimate_file_count(task)
    
    # 4. 判断是否需要UI
    needs_ui = requires_ui_design(task)
    
    # 5. 判断业务复杂度
    complexity = analyze_business_complexity(task)
    
    # 6. 决策
    if estimated_lines < 50 and estimated_files <= 2 and has_reference and not needs_ui:
        return "Level 2"
    elif estimated_lines < 200 and has_reference and complexity == "medium":
        return "Level 3"
    else:
        return "Level 4"
```

**Lead输出示例：**

```
Level 1: "这是简单的文案修改，我直接实现。"
Level 2: "这是小功能，参考XX实现，走简化流程（约10分钟）。"
Level 3: "这是中等功能，有历史参考，走标准流程（约20分钟）。"
Level 4: "这是复杂新功能，走完整流程（约25分钟）。"
```

### 示例

**示例1：** "把dashboard标题改成'仪表盘'"
- 评估：0项（文案修改）
- 判断：简单
- 行为：直接改代码

**示例2：** "修复bug：member角色能看到删除按钮"
- 评估：1项（单文件权限判断）
- 判断：简单
- 行为：直接修复

**示例3：** "bug报告加个优先级选择"
- 评估：✅ UI设计 + ✅ 多文件 = 2项
- 判断：中等
- 行为：询问用户

**示例4：** "修复bug：数据拉取后需要手动更新游标才能继续"
- 评估：✅ 产品澄清（根因不明） + ✅ 多文件 + ✅ 重大影响（核心逻辑） = 3项
- 判断：中等
- 行为：询问用户

**示例5：** "主页新增bug提交菜单和页面"
- 评估：✅ UI + ✅ 产品澄清 + ✅ 多文件 + ✅ 重大影响 = 4项
- 判断：复杂
- 行为：自动启动AI Team Workflow

## 角色（精简为4个agent）

| 角色 | 实现 | 说明 |
|------|------|------|
| **Lead** | 主 Claude 实例 | 唯一对外接口，负责所有 checkpoint |
| **PM** | 子 agent | 结构化需求 + 冲突检查 + 验收标准 → PRD |
| **UI** | 子 agent | 独立原型（standalone-demo.html，双击预览） |
| **Dev** | 子 agent | 技术设计 + 实现（在feature分支工作） |
| **QA** | 子 agent | 对照验收标准 + `requirements/_qa-rules.md` 逐条验证 |

**人类永远只和 Lead 说话。**

**人类参与点（仅2个节点）：**
- 节点一：PM澄清 + 冲突检查（多轮对话，Lead有疑问随时问）
- 节点二：UI原型预览确认（standalone-demo.html）

**AI内部协作（其余节点）：** Lead监督Dev/QA，不打扰人类

**人类验收：** PR合并到staging后，在staging环境网站验收（不在本地）

## 五个节点（精简版）

### 节点一：产品澄清

**角色：** PM agent

**输入：** 人类原始需求（`00-input.md`）

**PM职责：**
1. **结构化需求** — 将一句话需求扩展为完整PRD
2. **探索现有功能** — Glob/Grep搜索类似功能、相关路由、现有组件
3. **冲突检查** — 判断新需求是否与现有产品逻辑冲突
4. **输出验收标准** — 明确的checkbox列表，供QA和人类验收使用

**Lead行为：** 多轮对话澄清（不限问题数，有疑问就问）

**输出：**
- `01-prd.md` — 包含需求背景、功能设计、验收标准、冲突检查结果
- `01-questions.md` — 澄清问答记录（中大功能）

**冲突检查示例：**
- 新需求："添加数据导出功能"
- 现有功能：`/dashboard/data` 已有"下载CSV"按钮
- **冲突：** 功能重复
- **Lead行为：** 询问人类"现有下载功能是否满足需求？如不满足，新功能与旧功能如何区分？"

**验收标准格式：**
```markdown
## 验收标准

### 功能验收
- [ ] Dashboard侧边栏显示"Report Bug"按钮
- [ ] 点击进入bug提交页面
- [ ] 表单包含标题、描述、复现步骤三个字段
- [ ] 提交成功后显示成功toast
- [ ] 表单自动清空

### 技术验收
- [ ] POST /v1/bug-reports API返回200
- [ ] 邮件发送到team@humanbased.ai
- [ ] 邮件包含user_agent和current_url

### 边界验证
- [ ] Title超过200字符时显示错误
- [ ] Description为空时禁用提交按钮
- [ ] 网络失败时显示错误toast
```

### 节点二：UI原型

**角色：** UI agent

**输入：** `01-prd.md`

**UI职责：**
- 生成独立原型 `standalone-demo.html`（单HTML文件，双击预览）
- 真实tech stack（React + Tailwind）
- Mock数据，完整交互

**Lead行为：** 生成后通知人类预览，等待确认

**人类反馈：** 描述看到的现象（不诊断原因）

**Lead分类并指挥UI修改：**
- 如果人类说"颜色不对"，Lead判断是否需要修改
- 如果人类说"缺少XX按钮"，Lead指挥UI修复
- 确认后视觉锁定

**输出：** `02-prototype/standalone-demo.html`

### 节点三：开发实现

**角色：** Dev agent + QA agent（Pre-QA）

**输入：** `01-prd.md` + `02-prototype/standalone-demo.html`

**Lead行为：**
1. 创建 feature 分支（`feat/{slug}`）
2. 派Dev agent设计架构（输出开发计划）
3. **派QA agent做Pre-QA审查**（新增）
4. Dev实现功能
5. Lead监督（不需要人类批准开发计划）

**Dev职责：**
- 技术设计（接口契约、数据模型、组件树）
- 输出开发计划到 `03-dev-plan.md`
- 实现功能（Mock → 真实API）

**QA Pre-QA职责（新增）：**

在Dev开始编码前，QA提前审查开发计划：

1. **参考历史数据**
   ```bash
   # 查找类似功能的历史QA报告
   grep -r "bug.*report" requirements/*/04-qa-report.md
   # 分析上次类似功能发现的问题
   ```

2. **审查开发计划**
   - 技术方案是否有安全隐患（如：用户输入未转义）
   - 是否考虑了错误处理（网络失败、API错误）
   - 是否需要数据库migration（风险评估）
   - 是否有性能隐患（N+1查询、大循环）

3. **对照历史问题**
   - 上次类似功能发现了什么问题？
   - 这次开发计划是否避免了重复错误？
   - 是否有新的风险点？

4. **输出Pre-QA建议**
   ```markdown
   ## Pre-QA审查意见
   
   ### 参考历史
   - 类似功能：bug-submission-page (PR #130)
   - 上次发现：缺少错误处理（P1）
   - 建议：开发计划中增加try-catch
   
   ### 风险点
   ⚠️  用户输入未提及转义 → 可能有XSS风险
   ⚠️  邮件发送无重试机制 → 网络失败会丢失
   
   ### 建议
   1. 用户输入使用DOMPurify或框架内置转义
   2. 邮件发送失败时记录到日志，后续补发
   ```

5. **Dev根据建议调整**
   - 如果是严重风险 → 调整开发计划
   - 如果是建议性 → 记录，实现时注意

**好处：**
- 问题前移：在编码前发现风险，避免返工
- 历史学习：利用过去的经验，不重复犯错
- 成本低：只是审查计划（文字），不是审查代码

**输出：**
- `03-dev-plan.md` — 开发计划
- `03-pre-qa.md` — Pre-QA审查意见（新增）
- 实际代码修改

### 节点四：自动化验证

**角色：** QA agent

**输入：** 
- `01-prd.md` + `requirements/_qa-rules.md` + 实际代码
- `03-pre-qa.md`（如有Pre-QA审查意见）
- 历史QA报告（requirements/*/04-qa-report.md）

**QA工作流程：**

**步骤0：参考历史数据（新增）**

在开始检查前，先学习历史经验：

```bash
# 1. 查找类似功能
grep -l "bug.*report\|邮件" requirements/*/01-prd.md

# 2. 读取历史QA报告
cat requirements/bug-submission-page/04-qa-report.md

# 3. 提取历史问题
# - 上次发现了什么bug？
# - 哪些是重复出现的问题？
# - 哪些验收场景值得参考？
```

**输出历史对比：**
```markdown
## 历史数据参考

### 类似功能：bug-submission-page (2026-04-20)
**上次发现的问题：**
- P1：缺少网络错误处理
- P2：缺少表单重置逻辑
- P2：缺少测试覆盖

**本次检查重点：**
- ✅ 网络错误处理已有
- ✅ 表单重置逻辑已有
- ⚠️  仍缺少测试覆盖（重复问题）

**改进：** Dev学习到了错误处理的重要性
**退步：** 测试覆盖仍然缺失（建议提升为强制要求）
```

**QA职责：**

**本地代码层面检查（不需要运行）：**

1. **代码审查**
   - 类型安全（TypeScript类型、Python类型提示）
   - 边界处理（空值、超长、负数、特殊字符的代码逻辑）
   - 安全漏洞（XSS风险点、SQL注入、敏感信息泄露）
   - 性能问题（N+1查询、大循环、内存泄漏风险）
   - 错误处理（try-catch、网络请求的.catch、边界检查）
   
2. **回归风险评估**
   - 改动影响范围分析（修改了哪些共享模块、组件、API）
   - 潜在受影响功能清单（至少列3个可能受影响的现有功能）
   - 标记需要人工回归测试的场景
   
3. **测试覆盖检查**
   - 核心逻辑是否有单元测试（至少覆盖主流程）
   - 边缘案例是否有测试（空数据、错误响应）
   - 如无测试，标记"缺少测试覆盖"（P2）
   
4. **对照全局QA规则**
   - 逐条检查`_qa-rules.md`中的所有规则（代码层面）
   - 发现违反规则立即标记为P0

**输出staging验收清单（给人类）：**

5. **生成验收测试用例**
   - 对照PRD验收标准，转化为具体操作步骤
   - 补充边缘案例测试场景（空数据、超长输入、特殊字符、并发）
   - 错误场景测试（断网、API失败、权限不足）
   - 回归测试场景（受影响的现有功能）
   
**格式示例：**
```markdown
## Staging验收清单

### 主流程测试
1. 访问 /dashboard/bug-report
2. 填写标题："测试bug"，选择优先级"Urgent"
3. 预期：显示红色警告条"⚠️ Urgent bugs..."
4. 点击提交，预期：显示成功toast，表单清空

### 边缘案例
1. 标题输入201个字符，预期：显示错误提示
2. 不选择优先级直接提交，预期：使用默认值"General"
3. 断网提交，预期：显示友好错误提示

### 回归测试
1. 访问 /dashboard（检查导航菜单是否正常）
2. 测试登录功能（检查认证是否受影响）
3. 测试其他表单提交（检查是否有JS错误）
```

**Bug分类：**
- **P0（阻塞）** — 功能完全不可用、数据丢失、安全漏洞、违反全局QA规则
- **P1（严重）** — 核心功能受损但有workaround、性能问题、错误恢复缺失
- **P2（一般）** — 边缘场景、体验问题、缺少测试覆盖

**处理流程：**
- **P0** → Dev立即修复，QA重新验证
- **P1** → Dev修复或提供workaround说明，QA确认
- **P2** → 记录到bug log，可延后修复（需在PR描述中说明）
- **需求理解偏差** → 回节点一或二（Lead分析根因）
- **缺少测试覆盖** → 标记P2，建议补充但不阻塞发布

**回归测试要求：**

QA必须在报告中包含：

1. **改动影响分析**
   - 列出所有修改的文件
   - 标注哪些是共享模块/组件
   - 分析改动可能影响的范围

2. **潜在受影响功能清单**（至少3个）
   - 直接依赖改动模块的功能
   - 使用相同API/组件的功能
   - 关键业务路径（登录、核心操作）

3. **Staging验收清单中的回归测试场景**
   - 为每个潜在受影响功能，提供具体测试步骤
   - 标注优先级（P0关键路径优先测试）
   - 如改动>5文件或修改共享模块，必须包含全应用smoke测试步骤

**QA报告模板：**

```markdown
# QA验证报告

## 零、历史数据参考（新增）

### 类似功能：bug-submission-page (2026-04-20)
**上次发现的问题：**
- P1：缺少网络错误处理
- P2：缺少测试覆盖

**本次对比：**
- ✅ 网络错误处理：已改进
- ❌ 测试覆盖：仍缺失（重复问题，建议强制要求）

### Pre-QA审查意见落实情况
- ✅ Pre-QA建议添加try-catch → 已实现
- ⚠️  Pre-QA建议邮件重试机制 → 未实现（记录P2）

## 一、本地代码检查

### 1. 代码审查
- [ ] 无TypeScript/类型错误
- [ ] 无XSS风险（用户输入转义检查）
- [ ] 无敏感信息泄露（console.log、注释、硬编码密钥）
```

## 常见问题

### Q1: 什么时候走AI Team Workflow？

**A:** Lead自动判断。评估5个维度（UI设计、产品澄清、多文件、新功能、冲突可能），符合4-5项自动启动，符合2-3项询问用户，符合0-1项直接实现。

### Q2: 为什么移除Architect agent？

**A:** 技术设计由Dev agent负责更高效。Architect单独存在会增加一轮沟通（PM → Architect → Dev），实测节省~8k tokens和10分钟。

### Q3: 为什么不在本地验收？

**A:** 本地环境（dev server）与staging环境（真实部署）存在差异。在staging网站验收更接近真实用户体验，且避免"本地通过但staging失败"的情况。

### Q4: 原型为什么是standalone-demo.html？

**A:** 单HTML文件可直接双击预览，零依赖、零配置、零登录。人类可以立即看到UI效果，避免"需要启动dev server、配置环境变量、登录账号"等前置步骤。

### Q5: 多轮对话会不会浪费token？

**A:** 节点一的多轮对话是**方向投资**。方向错误的完美执行浪费更多token。实测中，节点一多问2-3个问题（+1k tokens），可以避免节点四回退重做（-8k tokens）。

### Q6: 如何处理紧急bug修复？

**A:** 紧急bug通常是0-1维度（修复现有功能，不涉及UI设计），Lead会判断为"简单任务"，直接修复而不启动完整workflow。

### Q7: QA验证时跳过冒烟测试，会不会遗漏bug？

**A:** 冒烟测试由staging验收覆盖。QA专注于对照验收标准和代码审查，staging验收专注于真实环境下的端到端测试。

### Q8: 如果用户对复杂度判断有异议怎么办？

**A:** 用户可以明确指示。例如"这个直接实现就行"（强制简单流程）或"这个走完整流程"（强制AI Team Workflow）。Lead会尊重用户的判断。
