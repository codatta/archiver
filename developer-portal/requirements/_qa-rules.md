# QA 规则库

每条规则来自真实 bug，由 Lead 在节点五/六后评估是否提升为全局规则。

格式：
```
## 规则 NNN：标题
- 来源：feature/xxx（YYYY-MM）
- 规则：具体要检查什么
- 验证方式：如何触发和验证
```

---

## 规则 001：UI 三态处理
- 来源：初始规则
- 规则：所有异步数据区域必须处理 loading / error / empty 三种状态，不得只处理成功态
- 验证方式：断网、返回空数组、模拟 API 500 时检查 UI 是否有对应提示

## 规则 002：Member 角色权限边界
- 来源：feature/member-permissions（2026-04）
- 规则：member 角色不得看到 invite、edit、delete 等操作控件，即使后端不报错
- 验证方式：用 member 账号登录，检查所有涉及权限的操作按钮是否隐藏

## 规则 003：API Key 安全显示
- 来源：初始规则
- 规则：API Key 默认脱敏显示，明文仅在用户主动操作后短暂展示，不得在列表页全量显示
- 验证方式：检查所有 API Key 展示场景，确认默认为 `sk-...xxxx` 格式

## 规则 004：表单提交防重复
- 来源：初始规则
- 规则：所有提交按钮在请求进行中必须 disabled，防止重复提交
- 验证方式：点击提交后立即再次点击，确认只发出一次请求

## 规则 005：敏感信息不得泄露
- 来源：初始规则
- 规则：禁止在console.log、注释、错误提示中输出密码、token、API key等敏感信息
- 验证方式：全局搜索console.log，检查是否有敏感字段；查看错误提示是否暴露内部信息

## 规则 006：用户输入必须验证和转义
- 来源：初始规则
- 规则：所有用户输入在存储前验证格式，在渲染前转义HTML，防止XSS
- 验证方式：尝试输入`<script>alert(1)</script>`，检查是否被转义；输入超长/特殊字符，检查是否有校验

## 规则 007：网络请求必须有错误处理
- 来源：初始规则
- 规则：所有API调用必须有try-catch或.catch()，网络失败时显示友好错误提示
- 验证方式：断网或模拟API 500，检查UI是否有错误提示而非白屏/控制台报错

## 规则 008：可选字段必须有合理默认值
- 来源：feature/bug-priority-levels（2026-04）
- 规则：所有可选表单字段必须有合理默认值，不得要求用户必填或显示undefined
- 验证方式：不填写可选字段直接提交，检查是否使用默认值

## 规则 009：回归测试清单必填
- 来源：QA优化（2026-04）
- 规则：QA必须列出"潜在受影响功能清单"（至少3个），并在staging验收清单中标注需要回归测试的场景
- 验证方式：检查04-qa-report.md是否包含回归风险评估章节和staging验收清单

## 规则 010：核心逻辑必须有测试
- 来源：QA优化（2026-04）
- 规则：业务逻辑（数据处理、权限判断、计算逻辑）必须有单元测试，UI组件可以没有
- 验证方式：检查是否有对应的.test.ts文件；如无测试，必须在bug log中标记"缺少测试覆盖"

## 规则 011：QA必须参考历史数据
- 来源：QA优化（2026-04-22）
- 规则：QA开始检查前，必须查找类似功能的历史QA报告，对比上次发现的问题，分析改进/退步
- 验证方式：04-qa-report.md必须包含"历史数据参考"章节，列出至少1个类似功能的对比

## 规则 012：Pre-QA审查开发计划
- 来源：QA优化（2026-04-22）
- 规则：Dev输出开发计划后，QA提前审查技术方案，发现安全/性能/错误处理风险，输出Pre-QA意见
- 验证方式：中大功能必须有03-pre-qa.md文件；小功能可选但建议有
